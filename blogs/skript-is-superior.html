Skript is awesome, I swear!

This is what Skript looks like for anyone uninformed.

1 on join:
2  send "Hello, world!" to the player
3  add 1 to {_variable}

Syntax is based on expressions, effect and nodes (and a few others...), which are used together to create a natural looking language. The features arn't that developed, there's no OOP, there's near zero functional support, there's not even a officially supported lambda, but if you want to do anything in Minecraft, Skript can do it with nicely made statements.

The allowed verbosity of Skript is something that also lures me in personally. I can be as inexplicit and explicit as I would like, expression syntax often allows for optional words that make your sentences make sense.

[all [[of] the]|the] [server] [non(-| )]op[erator]s

Take this syntax pattern for example. This probably makes zero sense if you have never seen Skript before, but bare with me. [word] means an optional; none or the syntax, (x|y) means a option between two syntaxes and [x|y] means none, or an option between the two syntaxes. This should now allow you to read the syntax above

This could be:

all of the server operators

or... it could be:

ops

Do you understand how great that is?, both express it perfectly well, but you can use filler words to just, make it make more sense.

The fact that Skript is a DSL (Domain Specific Language) makes it so much nicer to work with specifically Minecraft ideas. No strict typing, automatic conversions, inferring values. It really is awesome.

What are we even doing at this point. The problem with Skript is that it has the capabilities to create libraries and wrappers and more advanced things but we don't have the community that would actually do cool things with it.

Another thing that really stunt Skript's growth within developer communities is that every Minecraft developer has a precondition to automatically disregard anything Skript because of "Performance". This is ridiculous. Yes, if you write bad code, the performance will be bad. The community is filled with most people having Skript as their first programming language, therefore, most examples will have bad code, leading to bad performance... duh...

Skript doesn't need to be fast, actually.

There is an unreasonable need for performance nowadays, even when performance isn't an issue, like at all. Alot of people will ask the question in the Skript community, "Is my code fast? %insert code%". First of all, you should evaluate whether your code needs to be fast, if you have a command that sends a chat message, think, would a help command heavily impact your server in any way? I heavily doubt it. Second, TEST IT!. If it works and doesn't raise MSPT (milliseconds per tick) then you shouldn't think about it, and if you can do it 1000 times in a tick without raising MSPT, then you should really not think about it.

Most Minecraft based things will generally be bound with how fast the Java code is going, you will only face lag if you don't understand how to abstract problems. Do you need to loop all the blocks within a chunk to check whether a chunk has a block in it, or can you just save the location of the block and check it that way?

Not using the whole of Skript is also another pitfall people fall into. I see alot of people extracting the axis' of vectors and doing math on them like Skript doesn't have a perfectly good vector implementation that can do whatever you would want... Why are you even using a vector if you arn't doing operations with them.

Overall...

I think there's alot of great about Skript and it gets misrepresented by a bad side of the community who desperately have zero idea what they are doing. The rise of AI doing anything you want (even if it's absolutely awful at Skript) and the fact that Minecraft is the domain definitely helps.
